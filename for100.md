## 2. 가장 위험한 문제 TOP 5 & 즉시 수정 항목 매핑

### [가장 위험한 문제 TOP 5]
1. **미완성 캔들 실시간 감시로 인한 스퀴즈 돌파 신호 Repaint 현상**
   - *해결 계획:* `[Phase 1]` 완성 봉 기반 신호 판정 시스템 구축.
2. **실시간 캔들의 미성숙 거래량과 완성 캔들 평균 비교에 의한 진입 완전 차단 결함**
   - *해결 계획:* `[Phase 1]` 볼륨 분석 대상 인덱스를 `df.iloc[-2]`로 격리 조정.
3. **서버 오프라인/재설정 시 발생한 청산 PnL이 일일 손실 한도 제한(daily_pnl_usdt)에 누계되지 않는 현상**
   - *해결 계획:* `[Phase 3]` 앱 기동 시 `PnL Reconciler` 작동 및 동기화 결과 값의 메모리 즉시 가산 반영.
4. **10초 주기 최대 80개 종목 REST API 호출 병목으로 인한 API Rate Limit (Too Many Requests) 유출 위험**
   - *해결 계획:* `[Phase 4]` 웹소켓 스트리밍 도입 및 REST 스캔 범위 상위 25개로 하향 제한.
5. **트레일링 스탑 등 봇 사이드 청산 작동 시 기존 거래소 OCO 알고 주문 취소 확인 루틴의 엄밀성 부족**
   - *해결 계획:* `[Phase 2]` 거래소 알고리즘 주문 일괄 취소 트랜잭션 통합 및 안전망 보완.

### [즉시 수정 필요한 항목]
* **[즉시 조치 1]** `generate_signal` 실행 전 `df = df.iloc[:-1]` 슬라이싱을 적용하여 완성 캔들 데이터 기반 신호 체계로 고정.
* **[즉시 조치 2]** 거래량 돌파 필터(`USE_VOL_FILTER`)가 실시간 캔들이 아닌 `df.iloc[-2]` (마지막 완성 캔들) 기준으로 연산하도록 수정.
* **[즉시 조치 3]** `_sync_trades_to_csv_async` 실행 시 청산 거래 동기화 결과를 메모리 상의 `trader.daily_pnl_usdt` 및 로컬 통계 누적 합에 즉시 반영.

## 3. 단계별 디버깅 및 개발 실행 계획

### [Phase 1] 진입 로직 100점 달성 (완성 봉 신호 고정 및 필터 정상화)
- **목표:** 지표 Repaint 휩소 제거 및 거래량 MA 필터 우회 현상 완벽 차단.

#### 📋 Plan지침
1. **미완성 실시간 캔들 배제**: 15분 봉(또는 설정된 타임프레임)이 종가 기준으로 완전히 마감되기 전에는 노이즈 신호가 다수 유발됨. 지표 연산 전에 항상 미완성 캔들을 슬라이싱하여 원천 차단하는 알고리즘 설계.
2. **거래량 비교 지표 안정화**: 실시간 미성숙 거래량과 완성 캔들들의 Volume MA 값을 직접 대조할 경우, 거래량 급증 필터가 대부분 비정상적으로 참(True)이 되지 않아 진입이 영구히 차단됨. 비교 시점 인덱스를 전일/전봉 완성 봉 기준으로 고정하여 연산 무결성 확보.
3. **새 봉(New Bar) 기준 스캔 최적화**: 15분 주기 중 1회만 판단하도록 보완하여 API 및 연산 오버헤드 완화.

#### 🛠️ Act지침
1. `core/strategy.py`의 `generate_signal` 초입에서 파라미터로 입력받은 `df`에 대해 `df = df.iloc[:-1]` 슬라이싱을 수행하여 완성 봉 데이터만 남김.
2. 슬라이싱이 적용된 상태에서 `curr = df.iloc[-1]`은 자연스럽게 "마지막 완성 봉", `prev = df.iloc[-2]`는 "직전 완성 봉"이 됨. `vol_val = curr.get('volume', 0.0)` 및 `vol_ma_val = curr.get('vol_ma', 0.0)`을 통해 완성 봉 거래량을 MA와 올바르게 비교하도록 고정.

### [Phase 2] 청산 로직 100점 달성 (잔여 OCO 알고 오더 일괄 청소 보장)
- **목표:** 봇이 강제 청산(트레일링 스탑, 타임아웃 등) 시 잔여 OCO 익절/손절 대기 주문이 남아 반대 방향 포지션을 개설하는 사고 방지.

#### 📋 Plan지침
1. **주문 상태 비동기 완전 격리**: 거래소 사이드에서 설정된 conditional/OCO 주문은 일반 `cancel_all_orders` API 호출만으로는 취소되지 않을 수 있음. 따라서 거래소 전용의 알고리즘 주문 조회/취소 엔드포인트를 호출하는 독립 트랜잭션 필요.
2. **이중 클렌징 보장**: 포지션을 정리(`close_position`)하기 직전과 롤백 청산 수행 전, 심볼 기준 미체결 알고리즘/조건부 주문들을 식별하여 반드시 명시적으로 취소 처리.

#### 🛠️ Act지침
1. `core/exchange.py`에 `cancel_algo_orders(symbol)` 비동기 메서드를 구현하여 OKX `privateGetTradeOrdersAlgoPending` 및 `privatePostTradeCancelAlgos`를 연계 실행.
2. Binance `core/exchange.py` 역시 이에 대응하는 OCO 취소 루틴을 인터페이스 스펙에 맞추어 `cancel_algo_orders(symbol)`로 바인딩.
3. `close_position(symbol, side)` 실행 최상단에서 `cancel_algo_orders`를 명시적 호출하여 포지션 정리 전에 잔여 대기 오더를 완전히 해제.
4. `core/mock_exchange.py` 내 `MockOKXClient` 및 `MockBinanceClient`에도 `cancel_algo_orders` 더미 메서드를 탑재하여 Mock 테스트 시 예외 발생을 방지.

### [Phase 3] 상태관리 안정성 100점 달성 (PnL Reconciler 구축)
- **목표:** 봇 재기동 시 일일 손실 한도 검증 로직의 무결성 보장.

#### 📋 Plan지침
1. **메모리 복구 루틴 정의**: 로컬 봇의 프로세스가 비정상 종료되거나 재부팅되었을 때, 이전에 실행한 청산 거래가 손실 제한 누계(`daily_pnl_usdt`)에서 망각됨. 기동 즉시 로컬 거래 로그와 거래소 API 체결 내역을 복구/동기화하고 당일 KST 00:00:00 이후 청산 손익을 자동 복구하는 시스템 설계.
2. **실시간 가산 동기화**: 백그라운드 체결 동기화 시 수집되는 신규 청산 내역의 PnL을 메모리 상태 변수(`trader.daily_pnl_usdt`)에 원자적으로 누적 반영.

#### 🛠️ Act지침
1. `core/engine.py` 내에 `_reconcile_daily_pnl_async()` 메서드를 구축하여, 로컬 CSV 내역을 조회해 한국 시간(KST) 오늘 기준의 누적 청산 PnL 값을 계산하고 `self.trader.daily_pnl_usdt`에 주입.
2. `_sync_trades_to_csv_async` 동기화 로직 내에서 신규로 감지된 청산 거래(`category == "청산"`)의 타임스탬프가 오늘 KST 날짜와 일치하는 경우, `self.trader.daily_pnl_usdt`에 누적 가산 처리.

### [Phase 4] 실시간 안정성 100점 달성 (웹소켓 데이터 공급 체계 마이그레이션)
- **목표:** REST Polling 방식의 한계 극복 및 API 제한(Rate Limit 429) 완전 예방.

#### 📋 Plan지침
1. **웹소켓(WS) 하이브리드 캐시 구조 설계**: 다수 종목(최대 80개)에 대해 10초 주기로 REST API 폴링 시 빈번히 429 Too Many Requests가 발생함. 실시간 웹소켓 구독 채널(`ccxt.pro`)을 활성화하여 캔들 데이터를 실시간 수신하고 로컬 캐시 메모리에 upsert.
2. **REST Fallback 가드 구축**: 웹소켓 연결 단절이나 지연 시에만 REST 폴링으로 일시 회귀하는 이중화 구축.
3. **Mock 테스트 완벽 지원**: 테스트 러너(`harness.py --mock`) 환경에서는 실제 웹소켓 인스턴스가 exchange 속성을 갖지 않으므로 `AttributeError`가 발생함. 이를 우회하는 안전 가드 삽입.

#### 🛠️ Act지침
1. `core/websocket_client.py`에 `WebSocketClient`를 구성하여 `ccxt.pro.watch_ohlcv` 루프 실행. 수신 데이터는 `Scanner`의 `_ohlcv_cache` 및 `_ohlcv_cache_ts`에 업데이트.
2. `core/scanner.py`는 웹소켓이 구동 중이고 캐시에 데이터가 존재하면 REST 조회를 건너뛰도록 처리.
3. `core/websocket_client.py`의 `start()` 실행부 최상단에 `if not hasattr(self.client, "exchange"): logger.warning("[WS] MockClient 감지. 웹소켓 구독을 우회합니다."); return` 가드를 장착하여 테스트 무결성 보장.

## 4. 로직 검증 및 최종 테스트 시나리오

개선 코드가 배포된 후 다음 시나리오를 바탕으로 통합 QA 검증을 수행합니다.

1. **Repaint 검증:**
   - 15분 봉 형성 중 가격이 급격하게 돌파를 일으켰다 후퇴할 때, 실거래 진입 신호가 켜지지 않고 실제 완성 시점에만 정확히 진입 주문이 나가는지 백테스트 로그와 대조 검증.
2. **청산 후 잔여 오더 검증:**
   - 트레일링 스탑에 의해 봇이 시장가 청산을 수행한 직후, OKX/Binance 거래소 웹/앱에 접속하여 대기 중인 TP/SL 대기 주문(Algo Order) 갯수가 정확히 `0`개로 변경되었는지 확인.
3. **재기동 PnL 동기화 검증:**
   - 포지션 진입 후 봇을 강제 종료하고, 거래소에서 수동 혹은 OCO 체결로 포지션을 청산한 다음, 봇을 다시 켰을 때 대시보드 및 내부 daily_pnl 변수에 해당 손익이 오차 없이 반영되어 있는지 검사.
4. **Mock 하네스 검증:**
   - `python harness.py --mock`을 가동해 웹소켓 초기 설정 시 `MockOKXClient` / `MockBinanceClient` 관련 속성 오류 로그가 사라지고 `PASS` 완료되는지 확인.
